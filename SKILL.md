# Nana PRD Writing Skill

## 目标
产出符合以下特征的 PRD：

- 先证明“为什么值得做”，再说明“具体怎么做”
- 既能写复杂业务需求，也能写轻量规格型需求
- 有明确边界、有取舍、有收益验证方式
- 能直接服务产品、设计、研发、测试、合作方对齐

一句话概括：

**先把价值讲透，再把本期范围和落地规则讲清。**

---

## 一、适用范围

本 Skill 适用于以下 5 类需求：

1. **业务机会型**
   - 有明确用户问题、业务机会、竞品参考
   - 需要完整论证“为什么做”
   - 如：新功能、新玩法、多场景能力补齐

2. **商业化优化型**
   - 目标是提升会员、订单、转化、收入、渗透
   - 需要梳理转化链路与触点优化
   - 如：入口改造、权益承接、支付转化优化

3. **平台适配型**
   - 目标是完成系统能力适配、SDK接入、平台准入
   - 收益更多体现在“合作前提”和“上线必要条件”
   - 如：鸿蒙适配、系统媒体能力接入、语音助手接入

4. **规格定义型**
   - 需求不复杂，但规格、尺寸、状态、点击行为要求明确
   - 重点是接入方式、展示规则、状态定义、交付物
   - 如：桌面卡片、组件、轻量入口、固定规格页面

5. **定制改造型**
   - 基于公版能力做客户/品牌/预装定制
   - 重点是“改了什么、没改什么、依赖什么”
   - 如：客户定制版、品牌版、公版功能定向改造

---

## 二、你的写作风格定义

你的 PRD 不是单纯功能说明书，而是**产品决策文档**。

固定思路：

1. 需求为什么成立
2. 为什么现在做
3. 为什么本期这样做
4. 为什么先做这些、不做那些
5. 做完如何验证是否有效

你的明显风格特征：

- 业务分析驱动
- 用户场景先行
- 竞品分析后一定沉淀设计结论
- 擅长做 MVP 收敛
- 会明确写本期边界和不做项
- 复杂问题喜欢用表格整理
- 小需求偏“规格化、规则化、交付化”
- 重视收益和验证闭环

## 三、总原则

### 1. 先价值，后功能
正文不要一上来堆功能点。先回答：

- 为什么做
- 解决什么问题
- 对谁有价值
- 对业务有什么收益

### 2. 至少有一个“结论层”
所有背景、竞品、现状分析，最后都要得出结论：

- 设计启示
- 设计原则
- 本期结论
- 方案取舍

只堆材料，不算完整 PRD。

### 3. 复杂需求必须写“取舍”
如果需求复杂，必须明确：

- 理想方案是什么
- 为什么不能一次做完
- 本期做什么
- 本期不做什么
- 二期看什么

### 4. 小需求不强行写成长文
如果是平台接入、规格卡片、定制改造等小需求：

- 可以弱化长篇竞品分析
- 强化规格、差异、状态、接口、交付物

### 5. 量化优先
能写数字就不写泛泛判断。优先使用：

- 渗透率
- 留存
- 订单量
- 收入
- 转化率
- 覆盖用户规模
- MAU 增量
- 接口完成率
- 交付时间点

### 6. 功能描述必须可执行
写功能时至少覆盖：

- 触发条件
- 展示逻辑
- 点击行为
- 跳转逻辑
- 成功状态
- 失败状态
- 边界 case
- 默认值 / 首次态 / 异常态

### 7. 文风必须像“决策”和“落地”
多用：

- 当前存在…
- 核心问题是…
- 本期聚焦…
- 一期优先…
- 设计结论是…
- 暂不纳入本期…
- 异常情况下…
- 默认…
- 若…则…

少用：

- 可能可以
- 先这么看
- 大家觉得
- 后续再说
- 应该差不多

## 四、复杂度路由规则

写 PRD 前，先判断复杂度，再选结构。

### 1. 复杂需求
适用：
- 多场景
- 多端协同
- 多状态
- 有业务收益目标
- 需要 MVP 收敛
- 需要竞品/行业/主端支撑

典型结构：
1. 一句话总结（可选但推荐）
2. 需求背景
3. 竞品 / 主端分析
4. 产品方案 / 全景方案
5. MVP 范围 / 本期取舍
6. 目标 & 收益
7. 功能详情
8. 埋点
9. 实验 / 观测

### 2. 中轻量需求
适用：
- 功能边界较清晰
- 模块不多
- 收益路径比较直接
- 不需要展开很重的方案收敛

典型结构：
1. 背景
2. 目标 & 收益
3. 竞品 / 参考分析
4. 功能描述
5. 埋点
6. 实验（可选）

### 3. 小需求 / 规格型需求
适用：
- 平台适配
- SDK 接入
- 卡片接入
- 固定规格页面
- 定制改造

写法重点：
- 不强行做大篇论证
- 直接讲背景、约束、范围、差异、规则、交付物

## 五、类型路由规则

在复杂度之外，再判断需求类型。

### A. 业务机会型

#### 适用场景
- 新功能立项
- 大场景能力补齐
- 新玩法、新入口、新链路建设
- 有明显用户痛点和外部参考

#### 推荐结构
1. 一句话总结（问题 / 机会 / 方案 / 价值）
2. 需求背景
3. 竞品 / 主端分析
4. 设计结论
5. 产品方案
6. MVP 取舍
7. 目标 & 收益
8. 功能详情
9. 埋点
10. 实验

#### 写作重点
- 背景至少包含用户、平台、外部三个视角
- 竞品分析必须导出“设计启示”
- 本期边界一定明确
- 定量目标尽量显式

### B. 商业化优化型

#### 适用场景
- 会员转化
- 支付链路优化
- 权益感知提升
- 触点改造、商业化入口优化

#### 推荐结构
1. 背景
2. 当前核心问题
3. 竞品/行业参考
4. 转化链路分析
5. 优化方案
6. 目标 & 收益
7. 功能描述
8. 埋点
9. 实验方案

#### 写作重点
- 明确现有链路卡点
- 指出每个优化点对应的转化目的
- 收益拆成定性 + 定量
- 实验设计尽量标准化

### C. 平台适配 / SDK 前置型

#### 适用场景
- 鸿蒙适配
- 系统能力接入
- 车机系统联动
- SDK / API / 平台适配
- 合作准入能力建设

#### 推荐结构
1. 背景
   - 技术背景
   - 业务背景
   - 需求定位
2. 收益
   - 合作前提
   - 业务收益
3. 需求
   - 能力规划
   - 首期范围
   - 技术要求
   - 时间要求

#### 写作重点
- 强调“不做无法合作 / 无法上线 / 无法接入”
- 用表格说明平台差异
- 用优先级说明能力规划
- 用时间表说明里程碑和交付物

#### 常用表格
1. 平台差异表  
   字段建议：
   - 维度
   - 现有平台
   - 目标平台
   - 差异结论

2. 能力规划表  
   字段建议：
   - 能力模块
   - 说明
   - 优先级
   - 状态

3. 时间要求表  
   字段建议：
   - 节点
   - 日期
   - 交付物

### D. 规格定义 / 卡片接入型

#### 适用场景
- 桌面卡片
- 小组件
- 固定尺寸模块
- 轻量卡片式入口
- 外部容器接入内容模块

#### 推荐结构
1. 背景
2. 目标
3. 需求 / 接入约束
4. 规格概述
5. 卡片内容与功能定义
6. 卡片状态
7. 视觉设计交付物

#### 写作重点
- 强调接入约束和承载目标
- 重点写规格，不写过重背景分析
- 用表格说明尺寸、展示内容、按钮、点击行为、边界 case
- 状态逻辑必须单列
- 交付物必须明确

#### 常用表格
1. 规格概述表  
   - 规格
   - 尺寸
   - 说明

2. 内容定义表  
   - 类型
   - 内容

3. 点击行为表 / 合并字段  
   - 点击区域
   - 行为
   - 说明

4. 状态表  
   - 状态
   - 展示逻辑

### E. 定制改造 / 公版差异型

#### 适用场景
- 客户定制版
- 预装差异版
- 品牌版本
- 公版已有功能做定制包装

#### 推荐结构
1. 需求背景
2. 改造范围
3. 交互视觉说明（仅改造部分）
4. 文案替换
5. 接口依赖
6. 边界说明（可选）

#### 写作重点
- 先给公版 vs 定制版对照表
- 只写改动点，不重写公版全量逻辑
- 把入口、弹窗、参数、开关、接口、文案替换写清
- 若有视觉稿，以“相关功能和文案说明需保持一致”为约束

#### 常用表格
1. 改造范围对照表  
   - 改造项
   - 公版
   - 定制版

2. 文案替换表  
   - 位置
   - 公版文案
   - 定制文案

3. 接口依赖表（可选）  
   - 能力
   - 来源
   - 说明

## 六、标准结构模板库

### 模板 1：复杂业务型 PRD

#### 0. 一句话总结
建议四段式：

- 问题：
- 机会：
- 方案：
- 价值：

#### 1. 需求背景
建议按三层组织：

1. 用户视角
2. 外部视角（竞品/行业）
3. 平台视角（现状/能力基础）

要求：
- 有场景
- 有数据
- 有问题归纳
- 段尾有结论

#### 2. 竞品 / 主端分析
目标不是罗列信息，而是导出：

- 哪些能力被验证
- 哪些设计可借鉴
- 哪些能力不应照搬
- 一期该怎么做

#### 3. 设计结论
显式给出：

- 核心设计原则
- 本期聚焦场景
- 本期不覆盖内容
- 方案方向

#### 4. 产品方案 / 全景需求
先写理想方案，再写本期收敛。

#### 5. MVP 范围
固定要回答：
- 理想上希望做到什么
- 当前约束是什么
- 本期先做什么
- 本期不做什么
- 二期看什么

#### 6. 目标 & 收益
拆成：

- 定性收益
- 定量收益
- 若可能，补充测算依据

#### 7. 功能详情
按以下三种方式选一种：

##### 7A. 按阶段写
适合流程型需求：
- 进入前
- 使用中
- 结束后
- 异常中断

##### 7B. 按场景写
适合多入口需求：
- 场景一
- 场景二
- 通用规则
- 异常处理

##### 7C. 按模块写
适合商业化和页面优化：
- 模块名
- 需求描述
- 涉及页面
- 功能说明
- 跳转逻辑
- 状态规则

#### 8. 埋点
至少明确：
- 核心触发动作
- 转化节点
- 关键结果指标
- 参数归因要求

#### 9. 实验 / 观测
适合目标存在不确定性的需求：
- 实验目标
- 分组方式
- 核心指标
- 评估周期
- 决策规则

### 模板 2：中轻量 PRD

1. 背景
2. 目标 & 收益
3. 竞品 / 参考分析
4. 功能描述
5. 埋点
6. 实验（可选）

要求：
- 保留价值说明
- 简化全景方案和收敛过程
- 重点写功能规则和结果指标

### 模板 3：平台适配 / SDK 型 PRD

1. 背景
   - 技术背景
   - 业务背景
   - 需求定位
2. 收益
   - 合作前提
   - 业务收益
3. 需求
   - SDK 能力规划
   - 首期需求
   - 技术要求
   - 时间要求

要求：
- 重点突出“合作必要性”
- 技术要求写清适配范围、接口规范、兼容要求
- 时间要求必须落到交付物

### 模板 4：卡片 / 规格定义型 PRD

1. 背景
2. 目标
3. 接入需求
4. 规格概述
5. 内容与功能定义
6. 状态说明
7. 视觉交付物

要求：
- 重点是规格而非长篇论证
- 明确尺寸、展示、点击、状态、边界
- 交付物单独列出

### 模板 5：定制改造型 PRD

1. 需求背景
2. 改造范围
3. 交互视觉说明（仅改造部分）
4. 文案替换
5. 接口依赖
6. 边界说明（可选）

要求：
- 先差异对照，后展开说明
- 只写增量改造点
- 文案和接口必须显式列出

## 七、分析框架

写任何 PRD 前，先用这个问题框架想清楚。

### 1. 这件事解决什么问题
属于哪一类？

- 用户痛点
- 业务短板
- 商业化卡点
- 平台能力缺失
- 合作前置条件
- 规格/定制要求

### 2. 为什么现在做
原因可能来自：

- 数据已证明问题存在
- 竞品已验证
- 主端已有能力可复用
- 当前资源窗口合适
- 合作方强诉求
- 上线前置依赖明确

### 3. 这件事复杂在哪
从四个维度判断：

- 多场景
- 多端协同
- 状态同步复杂
- 商业 / 交付 / 接口链路复杂

### 4. 最小闭环是什么
必须回答：
- 哪个场景优先
- 哪个能力先不做
- 哪条链路必须先打通
- 哪个效果可以后补

### 5. 成功怎么定义
至少明确 1 类成功指标：

- 功能渗透
- MAU 增量
- 留存提升
- 转化提升
- 订单增长
- 收入增长
- 能力接入完成
- 合作准入达成
- 交付验收通过

## 八、收益写法规则

### 1. 定性收益
优先从四类写：

- 补短板
- 缩差距
- 强心智
- 打基础

常用句式：
- 补齐 XX 能力短板
- 缩小与竞品差距
- 提升用户对 XX 的感知
- 为后续 XX 能力扩展打基础

### 2. 定量收益
尽量写成：
- 指标
- 目标值
- 测算依据

例如：
- 功能渗透 2%
- 次留 +0.05pt
- XX 端 MAU 增量 +14.4w
- 会员转化率 +x%
- 点击率 +x%
- 接口联调按期完成

### 3. 平台 / 适配类收益的表达方式
如果收益并非直接体现在 C 端指标，可以写：

- 合作前提收益
- 上线准入收益
- 覆盖车型 / 平台收益
- 平台能力补齐收益
- 后续扩展基础收益

---

## 九、MVP 取舍规则

这部分是你的核心能力。复杂需求必须显式写。

### 标准写法
1. 理想态
2. 现实约束
3. 本期范围
4. 不做项
5. 二期方向

### 常用句式
- 理想上…
- 结合研发资源和实现复杂度…
- 本期核心定位是…
- 一期优先补齐…
- 本期暂不支持…
- 二期再考虑…

### 禁忌
- 不要只写“后续可扩展”
- 不写清“不做项”，项目很容易发散

- 开启 / 关闭
- 成功 / 失败 / 异常
- 播放中 / 已暂停 / 空态 / 无网

## 十一、表格优先规则

你很适合高密度表格表达。以下内容优先表格化：

- 平台差异
- 竞品对比
- 能力规划
- 优先级
- MVP 范围
- 卡片规格
- 状态说明
- 文案替换
- 时间节点
- 接口依赖

原则：
- 复杂内容先表格归纳，再补文字说明
- 表格负责“收束”，正文负责“解释和结论”

## 十二、默认句式库

### 1. 背景句式
- 当前…存在以下核心问题
- 从用户视角看…
- 从平台现状看…
- 从竞品和行业趋势看…
- 现阶段不支持…导致…

### 2. 结论句式
- 综合分析，设计结论如下
- 因此本期优先…
- 结合当前资源和复杂度…
- 核心策略是…
- 本期范围聚焦于…

### 3. 功能句式
- 点击后…
- 若…则…
- 默认…
- 异常情况下…
- 当…时展示…
- 无…时不展示…
- 成功后…
- 返回时…

### 4. 边界句式
- 本期暂不涉及…
- 本期不支持…
- 二期再考虑…
- 以下能力不纳入当前版本…

### 5. 交付句式
- 交付物包括…
- 预计时间节点为…
- 联调完成标准为…
- 上线前需完成…

## 十三、禁忌清单

### 1. 禁止一上来写功能清单
没价值、没背景、没结论，PRD 会很弱。

### 2. 禁止只有竞品，没有结论
竞品只是材料，不是答案。

### 3. 禁止只写“用户有需求”
要回答：
- 需求有多大
- 是否高频
- 是否值得做
- 为什么现在做

### 4. 禁止复杂需求不写边界
不写范围和不做项，项目会发散。

### 5. 禁止规格型需求写成空泛长文
卡片、接入、定制类文档，重点是规则，不是抒情。

### 6. 禁止定制需求重写公版全量逻辑
只写差异和增量。

### 7. 禁止目标没有验证方法
没有指标、埋点、验收方式，收益无法复盘。

## 十四、成稿检查清单

### A. 价值层
- [ ] 是否说明了为什么做
- [ ] 是否明确了问题和目标对象
- [ ] 是否有收益描述
- [ ] 是否有至少一个量化目标或可验证结果

### B. 决策层
- [ ] 是否有设计结论
- [ ] 是否写清本期范围
- [ ] 是否写清不做项
- [ ] 是否说明了取舍理由

### C. 落地层
- [ ] 是否写清入口、展示、点击、跳转
- [ ] 是否覆盖关键状态
- [ ] 是否有异常处理
- [ ] 是否写清边界 case
- [ ] 是否给出交付物 / 接口 / 时间要求（如适用）

### D. 验证层
- [ ] 是否有埋点思路
- [ ] 是否有实验或观测方案（如适用）
- [ ] 是否能判断上线成功与否

## 十五、输出策略

当接到新需求时，按以下顺序工作：

1. 先识别需求类型  
   - 业务机会型 / 商业化优化型 / 平台适配型 / 规格定义型 / 定制改造型

2. 再识别复杂度  
   - 复杂需求 / 中轻量需求 / 小需求

3. 选对应结构模板

4. 自动补齐以下内容  
   - 为什么做
   - 本期范围
   - 关键规则
   - 收益或交付价值
   - 验证 / 验收方式

5. 输出时遵循以下文风  
   - 结论前置
   - 表格收束
   - 语言明确
   - 边界清楚
   - 不写空话

## 十六、你的个人方法论一句话

**复杂需求，讲清价值和取舍；小需求，讲清规格和边界。**

---

## 十七、可持续迭代方式

后续优化本 Skill 时，按以下顺序演进：

### v1
- 固化骨架
- 明确总原则
- 建立复杂度路由

### v2
- 补齐类型路由
- 增加小需求分支
- 增加模板库和句式库

### v3
- 补优秀案例库
- 为每类需求增加“正例”
- 沉淀高频表格模板

### v4
- 补反例库
- 明确哪些写法你不喜欢
- 强化“删繁就简”的判断规则

### v5
- 行业细分
- 车载/IOT/商业化/合作定制分别形成专用分支

---

## 十八、最终输出要求

最终产出的 PRD，应满足：

- 可以给老板看：有业务判断
- 可以给设计看：有页面与交互规则
- 可以给研发看：有范围、状态、接口、时间要求
- 可以给测试看：有边界和异常逻辑
- 可以给合作方看：有明确对齐项和交付物

如果只能满足其中一类对象，说明 PRD 还不够完整。
